## Collection: `events`

Top-level collection. One document per posted event.

### Document fields

| Field         | Type                    | Required | Notes |
|---------------|-------------------------|----------|-------|
| `title`       | `string`                | yes      | 1–80 chars. Trimmed on write. |
| `description` | `string`                | yes      | 1–500 chars. Trimmed on write. |
| `location`    | `string`                | yes      | MUST match a key in `app/constants/locations.ts` (D3). This is the join key for filtering events to a tapped pin (B4). |
| `clubName`    | `string`                | yes      | 1–60 chars. The org or person hosting. |
| `startTime`   | `Timestamp` (Firestore) | yes      | When the event begins. Display-only; not used for the active filter. |
| `endTime`     | `Timestamp` (Firestore) | yes      | When the event ends. **This is the `expiresAt` field** the active-events query (A4) filters on. |
| `createdAt`   | `Timestamp` (server)    | yes      | Set with `serverTimestamp()` on insert. Never written from the client. |
| `createdBy`   | `string \| null`        | no       | Firebase Auth `uid` of the poster, or `null` for anonymous. |

### Document ID

Auto-generated by Firestore (`addDoc`). Do not roll a custom ID scheme.

### Why no separate `expiresAt` field

The task spec (A4) says "filters by `expiresAt > now`". In this schema
`endTime` *is* that field — an event is active until the moment it ends.
Adding a redundant `expiresAt` mirror invites drift between the two. If
we ever need a post-end grace period (e.g. show events for 1 hour after
they end), do it as a query-time offset, not a second stored field.

### Why `location` is a denormalized string

Pins live in `app/constants/locations.ts` (D3). Storing the full location
string on each event keeps reads cheap (no join) and matches how
`EventList` and `pinDetails` already render. Tradeoff: renaming a
location requires a one-shot migration. Acceptable for a class project
with no admin tooling — and the `LOCATIONS` constant stops typos at the
client.

## Indexes

Firestore auto-creates single-field indexes. Composite indexes have to
be declared.

| Index                              | Used by                       | Status |
|------------------------------------|-------------------------------|--------|
| `endTime` (ASC)                    | `getActiveEvents()` (A4)      | auto-created |
| `location` (ASC) + `endTime` (ASC) | per-pin event filter (B4)     | **add when B4 wires live data.** Firestore prints the create-index URL on the first failed query — click it once and commit no code. |

## Write path: `addEvent(input)`

`app/services/eventService.ts` (A3) is the only place client code should
write to `events`.

```ts
type EventInput = {
  title: string;
  description: string;
  location: string;   // must be a LOCATIONS key (D3)
  clubName: string;
  startTime: Date;
  endTime: Date;
};

async function addEvent(input: EventInput): Promise<string> {
  // 1. Validate (mirrors AddEventModal):
  //    - title 1–80, description 1–500, clubName 1–60
  //    - location non-empty AND in LOCATIONS
  //    - startTime > now, endTime > startTime
  // 2. Convert Date → Timestamp.fromDate(...)
  // 3. addDoc(collection(db, "events"), {
  //      ...validated,
  //      createdAt: serverTimestamp(),
  //      createdBy: auth.currentUser?.uid ?? null,
  //    });
  // 4. Return the new doc id.
}
```

Validation duplicated between the form (`AddEventModal`) and the service
is intentional: the form fails fast with field-level UI; the service is
the last line of defense against bad calls from anywhere else.

## Security rules (planned, not yet deployed)

Current state is test mode (open read/write). Before May 10, deploy
something like:

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /events/{eventId} {
      // Anyone can read.
      allow read: if true;

      // Signed-in users can create valid documents.
      allow create: if request.auth != null
        && request.resource.data.title is string
        && request.resource.data.title.size() > 0
        && request.resource.data.title.size() <= 80
        && request.resource.data.description is string
        && request.resource.data.description.size() <= 500
        && request.resource.data.location is string
        && request.resource.data.clubName is string
        && request.resource.data.startTime is timestamp
        && request.resource.data.endTime is timestamp
        && request.resource.data.endTime > request.resource.data.startTime
        && request.resource.data.endTime > request.time;

      // No client-side edits or deletes for v1.
      allow update, delete: if false;
    }
  }
}
```

## Out of scope for v1

- User profiles, RSVPs, or attendance counts.
- Editing or cancelling events. (Typos → repost.)
- Event categories, club search, image attachments.

Anything here is a future schema version, not a hot patch.
